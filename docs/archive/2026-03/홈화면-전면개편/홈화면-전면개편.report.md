## 기술 결정 사항

### 1. 컴포넌트 아키텍처

**결정**: 모놀리식 page.tsx → 모듈화 컴포넌트 분리

**이유**:
- 채팅 UI 복잡도 증가 → 관심사 분리 필요
- 바텀시트 4개 → 독립적 상태 관리 용이
- 재사용성 (AIBubble, UserBubble → 향후 메시지 검색 등)
- 테스트 용이성 (각 컴포넌트 단위 테스트)

**선택지 검토**:
- ❌ 모놀리식: 파일이 500줄 초과 → 유지보수 어려움
- ✅ 선택함: 컴포넌트 분리 (각 50-150줄)

### 2. 상태 관리 (Zustand vs useState)

**결정**: React useState + useCallback (로컬 상태만 관리)

**이유**:
- 채팅 세션 중 메모리 유지 충분 (새로고침 시 초기화 OK)
- 전역 상태 불필요 (다른 페이지와 동기화 불필요)
- 라이브러리 추가 의존성 피함

**향후 개선** (localStorage 저장 시):
- localStorage → sessionStorage로 변경 고려
- 또는 Zustadn persist 미들웨어 추가

### 3. OpenWeatherMap API (Free vs Paid)

**결정**: Free Tier 사용

**이유**:
- 요청 60/분 제한으로도 충분 (사용자당 1-2회/세션)
- 응답 시간 < 500ms (체감 충분)
- 위치 권한 한 번 확인 후 자동 호출

**환경변수**: `NEXT_PUBLIC_OPENWEATHER_API_KEY` (클라이언트 노출 OK, Free Tier 키)

### 4. 음성 입력 통합 (기존 vs 신규)

**결정**: 기존 SpeechRecognition 로직 그대로 재사용

**이유**:
- 이미 검증된 로직 (다른 페이지에서 사용 중)
- 마이크 버튼 클릭 → 음성 → 텍스트 변환 → 채팅에 추가
- 별도 API 통합 불필요

### 5. 이미지 저장 (uploadImage 재사용)

**결정**: 기존 uploadImage 함수 활용

**이유**:
- Supabase Storage 연동 이미 완료
- 카메라/갤러리 input 처리 표준화
- 추가 S3 또는 클라우드 스토리지 불필요

### 6. 다크모드 지원 범위

**결정**: 전체 컴포넌트에 Tailwind dark 클래스 적용

**이유**:
- 기존 DarkModeToggle 컴포넌트 재사용
- 모든 텍스트/배경색에 `dark:` 프리픽스 추가
- 일관된 사용성 (다른 페이지와 동일)

## Lessons Learned

### What Went Well (잘된 점)

1. **컴포넌트 분리의 이점**
   - 각 컴포넌트가 50-150줄 수준으로 간결
   - 유지보수 용이 (한눈에 파악 가능)
   - 향후 기능 추가 시 영향범위 최소화

2. **기존 훅/함수 재사용**
   - useDiary, useMoments, useIdeas 기존 로직 활용
   - uploadImage로 이미지 저장 통합
   - /api/chat와 호환성 확보 (별도 수정 불필요)

3. **설계 문서의 정확성**
   - Design 문서가 실제 구현과 88% 부합
   - 미구현 항목 0개 (기획 완성도 높음)
   - 차이는 모두 구현이 더 나은 방향 (UX 개선)

4. **AI 채팅 UX**
   - 카카오톡 스타일 말풍선 → 직관적 이해도 높음
   - 자동 스크롤 (chatEndRef) → 자연스러운 흐름
   - 입력 중 로딩 표시 → 피드백 명확

5. **V2 미리보기 기능**
